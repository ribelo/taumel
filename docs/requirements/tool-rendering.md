# Tool rendering

## Intent

Every Taumel tool call and result renders in the Pi TUI with a collapsed and an
expanded form. Collapsed clips to essentials, usually one line; expanded is rich
and recovers full content. Color is semantic only, with no syntax highlighting,
reusing existing theme tokens.

This requirement set covers every Taumel-emitted message-like surface that can
appear in the transcript or model-visible conversation stream: registered tool
calls, registered tool results, and custom messages emitted through Pi messaging
such as `notification` and `skill`. It does not cover non-message UI components
such as manager/editor screens, the footer, or composer UI.

Pi message handling has separate audiences: the agent-facing payload, the
user-facing compact render, and the user-facing expanded render. The
agent-facing payload is out of scope for this requirement set except where it
supplies structured facts to render. These requirements govern only what the
user sees in compact and expanded transcript rendering.

These requirements describe user-visible behavior, not a test strategy or an
implementation checklist. Verification should prove the behavior from a user's
point of view rather than assert incidental implementation details.

Following the shared engineering standards, known structured message types
should have explicit renderable shapes rather than ambiguous generic fallbacks
that can produce unreadable domain output such as anonymous `item N` rows.

## Requirements

### Shared grammar

- The system shall render every tool call with a one-line header `• <name> · <subject>`, the tool name in `toolTitle`, a dim separator, and a subject truncated with `…` when collapsed. ^render-hd01
- The system shall use the tool's snake_case name as the header name token for tool invocations and a lowercase dotted or snake_case message-kind token such as `goal.continue`, `cron.fire`, `exec_completion`, or `agent_completion` for custom messages; header name tokens shall not contain spaces or Title Case words. ^render-xgvf
- The system shall limit the header subject to call identity and shall place state and count facts in a dim trailing parenthesized group, joining facts inside parentheses with `, ` and reserving ` · ` for header segments. ^render-z747
- The system shall not add an outer transcript gutter to lines emitted by a Taumel tool or custom-message renderer; Pi owns the component/widget margin and its configured tool indentation. Taumel renderers shall use the width they receive and shall reserve indentation only for semantic internal structures such as body rails and diff gutters. ^render-hd09
- Compact and expanded renders of the same message shall keep the same primary header identity. Expanded renders may move additional facts into the body, but shall not substantially rename or reframe the header compared with compact rendering. ^render-hd08
- The system shall color the header dot by state: green for ok or exit 0, red for failure or non-zero exit, and yellow for running. ^render-hd02
- For aggregate results containing multiple item outcomes, the header dot shall represent the aggregate user outcome: green only when every relevant item succeeded or is healthy, red when any relevant item failed, was cancelled/timed out/lost, or otherwise did not succeed, and yellow only for still-running/in-flight states. ^render-hd04
- For run-like aggregate statuses, `completed` shall be green; `running`, `queued`, `suspended`, and bounded waits that timed out while work remains live shall be yellow; `failed`, `cancelled`, `timed_out`, and `lost` shall be red. Mixed aggregates shall use the highest-severity contained status: red over yellow over green. ^render-hd05
- When a call is in flight, the system shall render a yellow dot and a trailing dim `(<progress>)`, and shall not repeat the exit code in the subject. ^render-hd03
- In-flight tool calls shall render as a single compact line with a yellow dot, tool name, meaningful subject, and useful trailing progress text such as `(running)`, `(waiting)`, or `(viewing image)`. In-flight calls shall not render a body. ^render-hd06
- The system shall hang the body under the header with a dim `└` connector and align continuation lines with a 4-space indent, in `toolOutput`; structured aligned bodies such as diffs may render flush without the connector to preserve their own gutters. ^render-bd01
- The system shall show truncation as a single dim `… N more` line, placed at the top for tail-oriented tools and the bottom for head-oriented tools, without hardcoding the expand keybinding. ^render-bd02
- The system shall render empty output as dim `(no output)`, or `(none)` for empty lists. ^render-bd03
- The system shall omit a separate metadata block, carrying state in the dot and load-bearing facts in the header's trailing facts group. ^render-bd04
- The system shall render truncated plain-text tool output on complete output-line boundaries whenever at least one complete line fits the display budget. If an individual line is too long to fit, the system may show a bounded UTF-8-boundary-safe long-line preview only with explicit truncation metadata or an omission marker; it shall never show a silent mid-line slice. ^render-bd05
- The system shall use tail-oriented truncation for command execution output and head-oriented truncation for file/read/search-like bodies. Head-oriented truncation shall not render a partial first line; when the first line exceeds the byte budget, it shall render an explicit recovery message naming the line, limit, and how to inspect the full content instead. ^render-bd06
- Expanded labeled fields shall use the form `Label: value` with a dim Sentence-case label and a `toolOutput` value, shall capitalize `ID`, and shall not render labeled data as `key=value` pairs or unlabeled bare values. ^render-6g15
- The system shall render omission and truncation with the single-character ellipsis `…`, shall render transformation arrows as `→`, shall separate segments with single spaces, and shall not terminate headers, facts, or labels with a period. ^render-zcag
- The system shall register every Taumel-rendered model-callable tool with `renderShell = "self"` so Pi's default tool shell cannot background-paint multiline bodies, diff gutters, indentation, wrapped whitespace, or blank rows. ^render-sh01
- The system shall keep every Taumel-rendered tool line below the terminal's last column when width permits, leaving a one-cell guard so exact-width writes cannot trigger terminal auto-wrap or smear background color after wrapped fragments. ^render-sh02
- The system shall clip collapsed renders and show full content-bearing fields when expanded, with no line cap and no per-line truncation. ^render-ce01
- The system shall use color only for the state dot (`success`, `error`, `warning`, or `muted` for stateless messages), diff added and removed lines, dim separators, labels, trailing facts, and counts, `toolOutput` body text, `accent` ordinal indices, and Pi-parity skill rendering, with no syntax highlighting, and shall introduce no new theme tokens. ^render-cl01
- Taumel custom message renderers shall render flush without a Pi background box, matching self-shell tool chrome; a Pi-owned parity renderer such as `skill` shall keep Pi's own framing. ^render-noby
- The system shall keep compact renders to a single physical line by default. Compact multiline previews are allowed for side-effecting/action tools where a small preview is the user's quickest signal of what changed or whether the action succeeded, including `exec_command`, `write_stdin`, `edit`, `write`, and `apply_patch`. ^render-ux01
- The expanded render shall be maximally informative for the user without being maximally verbose: it shall show the useful human-facing status, identifiers, and content needed to understand what happened, while omitting model-only implementation details. ^render-ux02
- User-facing renders shall be human-readable and shall not expose raw JSON, XML, or protocol envelopes unless the user explicitly asked to inspect JSON/XML as the payload content itself. ^render-ux03
- Optional fields shall appear in user-facing renders only when they carry information: omit absent, null, empty-string, empty-list, and false boolean fields unless false is itself the important state being reported. ^render-ux05
- Compact rendering shall never rely on terminal wrapping. Every compact physical line, including allowed compact multiline preview rows, shall be explicitly clipped or abbreviated to the available renderer width with an omission marker when needed. Expanded rendering may wrap long human-readable fields, except for structured aligned content such as diffs where wrapping would destroy readability. ^render-ux06
- Compact rendering of long paths and long opaque identifiers shall prefer middle truncation so both the meaningful prefix/location and suffix/filename/extension remain visible, e.g. `/tmp/pi-clipboard-6e6a…501780841a8c.png`, rather than hiding the suffix with right-edge clipping. ^render-ux07
- Expanded rendering of paths and opaque identifiers shall preserve the complete value. The complete value may wrap across physical lines, but it shall not be silently truncated. ^render-ux08
- Failed tool results shall render with a red dot and a useful compact error summary. Single-line compact tools shall include the short error reason in the subject or trailing facts when possible; compact multiline exceptions may show a short error body line when the reason does not fit the header. Expanded failure rendering shall show the full error text plus relevant user-facing context, with no raw JSON/XML envelope. ^render-er01
- Expanded list/entity renders shall use a hybrid readable style: a concise headline row for each item/entity, followed by labeled secondary fields only when there are multiple details or long text. Dense headline rows shall not include long descriptions; descriptions belong on secondary lines. The system shall avoid giant table layouts unless alignment materially improves readability. ^render-2rsg
- Expanded lists shall insert blank lines between items when items contain secondary lines such as descriptions, prompts, snippets, or labeled metadata. Compact renders shall not add blank separator lines, and expanded dense one-line-only lists may omit blank separators. ^render-463z
- Known structured message domains, including profiles, agents, cron tasks, goals, threads, Exa results/runs/events, notifications, and skills, shall normalize into explicit renderable domain objects before layout. ^render-dm01
- Taumel shall not maintain a competing pseudo-generic fallback renderer for structured domains. Pi provides per-slot fallback rendering for registered tools whose `renderCall` or `renderResult` is absent or throws: calls fall back to the tool name and results fall back to raw text content. That Pi raw-text fallback is acceptable as a resilience path so the application does not crash, but it is not the intended rendering path for Taumel-owned tools. ^render-dm02
- Every Taumel-owned tool shall have an explicit user-facing renderer, even if that renderer deliberately chooses a minimal raw-text presentation for a simple tool. Missing or failing renderers shall degrade to Pi's raw-text fallback rather than crashing, but missing renderers are not considered complete for Taumel tools. ^render-dm03
- `renderShell = "self"` shall be used only when Taumel supplies the tool's own renderer/framing. A tool without a Taumel renderer shall not force self-shell rendering; it shall let Pi's fallback shell own the framing. ^render-dm04
- An explicit renderer may be intentionally minimal for simple tools or actions. Renderer completeness means the presentation choice is deliberate and human-readable, not that every tool must have a rich multiline expanded view. ^render-dm05

### Per-tool

- The system shall render `exec_command` and delta-mode `write_stdin` as compact multiline exceptions because command output is often the user's fastest success/failure signal. Compact shall use the command, sent chars, or `poll session N` as subject; include running/session/exit state in the header when available; and show a tail-oriented preview of roughly the last 5 output lines, clipping compact physical lines rather than wrapping them. Expanded shall show the full returned output for that tool result, stripping ANSI and written-echo where applicable. A running async session shall use a yellow dot with `(session N)`, and an empty delta poll shall show `(still running, no new output)`. Status-mode `write_stdin` shall remain one concise status line in compact and expanded views, showing running or terminal state plus suppressed output counts and no stdout/stderr preview. ^render-ex01
- The system shall render `read` compact as a single line, using `• read · <path> (lines <start>–<end> of <total>)` for a result that returns fewer than the total file lines and `• read · <path> (<N> lines)` for a complete-file result, and shall render it expanded as exactly the returned text content with its line-number gutter and any truncation/recovery footer such as `… N more`, without adding a separate metadata block. ^render-rd01
- When a `view_media` call is in flight, the system shall render a yellow-dot header with trailing progress text `(viewing image)`. ^rendermedia-a8k2
- The system shall render successful `view_media` results with the path and processed dimensions in the subject, using `<original-width>x<original-height> → <width>x<height>` when the image was resized. ^rendermedia-h6m4
- The system shall render `view_media` compact as a single line with path and dimensions, middle-truncating long compact paths when needed. Expanded, it shall show the full path, MIME/type when known, original dimensions, processed dimensions, whether resizing occurred, payload size when available, and the tool-result text summary. User-facing render output shall not show image base64. ^rendermedia-p9x1
- The system shall render `edit` and `apply_patch` as compact multiline exceptions using a real unified diff (via the `diff` package) with a dim right-aligned line-number gutter, a `(+N -M)` subject summary, 2 context rows above and below the change block capped near 6 changed lines collapsed, and full hunks expanded. Added and removed lines shall color the whole changed content line with existing added/removed diff theme colors, not only the `+`/`-` marker; context lines shall remain normal/dim. Compact diff lines shall clip rather than wrap, and expanded diffs may also clip aligned diff lines to preserve structure. No word-level diff or diff background coloring is required for v1. `apply_patch` with a single write and no deletes renders like `edit` when collapsed. A multi-file `apply_patch` shall render an aggregate header followed by a dim `└` connector, a white `path (+x -y)` heading, and the same compact line-numbered diff preview used for a single-file patch for every affected file, with one empty line between file sections in both collapsed and expanded views; a deleted file shall show its previous contents as removed diff lines rather than only a `(deleted)` summary. Expanded always shows each file's full diff. ^render-df01
- When `apply_patch` fails, the system shall render the failure reason to the user instead of only a red dot: collapsed shows a short error line and expanded shows the full tool-result/error text. ^render-df02
- The system shall render `write` as a compact multiline exception: compact shall show path, a mode and line-count summary that states line units in both modes, and a tightly capped preview of the written content head such as 3 lines plus `… N more`. Expanded, it shall show full path, mode (`overwrite`/`append`), byte and line counts when available, and the full written content without raw JSON. ^render-wr01
- The system shall render collection tools (`web_search_exa`, `crawling_exa`, `exa_agent_list_runs`, `exa_agent_list_events`) compact as a single-line query/filter subject plus count/state summary, with no compact item rows. Expanded, it shall show useful item rows up to the tool's reasonable display cap, with optional secondary lines for URLs, descriptions, summaries, or status details, and `(none)` when empty. ^render-co01
- The system shall render `query_threads` compact as a single line `• query_threads · "<query>" (N threads, M hits)` with no compact thread rows. Expanded, it shall group by thread and show title/id, workspace when known, bounded summary metadata when present, and top hits with kind/role/tool/timestamp plus snippet, with no raw JSON and no score numbers. ^render-th01
- The system shall render `read_thread` compact as a single line `• read_thread · <threadID> (<mode>[, facts])`, including concise facts such as message count, diagnostics count, or `more available` when present, with no compact metadata row. Expanded, it shall render structured thread content by mode: `overview` shows metadata, summaries, and recent visible entries; `window` shows target metadata plus adjacent visible entries and marks the target position textually; `full` shows the transcript page plus cursor/truncation footer. Expanded may include title, workspace, mode, cursor, and diagnostics when present, and shall render no raw JSON. ^render-th02
- The system shall render single-entity and action tools with a header plus one dim `·`-joined facts line collapsed, the header alone when there is nothing extra, and the full text body appended uncapped when expanded. ^render-se01
- The system shall render `cron_create` compact as a single-line task summary containing task id, schedule, and enabled state. Expanded, it shall show task id, raw cron expression, human schedule, mode, recurring versus one-shot, enabled state, next due time, and prompt, with no raw JSON. ^render-cr01
- The system shall render `cron_list` compact as a single-line summary containing task count and cron master switch state, e.g. `3 tasks (enabled)` or `3 tasks (disabled)`, without per-task compact rows. Expanded, it shall show the master switch state and every task with id, schedule, mode, recurring/one-shot, enabled state, next due time, and prompt, with no raw JSON. ^render-cr02
- The system shall render `cron_delete` compact as a single-line deletion outcome, e.g. `<id> (deleted)` or `<id> (not found)`. Expanded, it shall show the id and deletion outcome, with no raw JSON. ^render-cr03
- The system shall register an explicit renderer for `taumel.cron.fire`. Compact, it shall render one physical line identifying a cron fire with task id, human-readable schedule, and coalesced count when greater than one. Expanded, it shall show the task id, raw cron expression, human-readable schedule, coalesced count, and complete scheduled prompt, with no raw JSON or protocol envelope. ^render-cr04
- Live and replayed `taumel.cron.fire` messages shall use the same renderer and shall remain visually distinct from user messages, ordinary assistant text, and completion-availability notifications. ^render-cr05
- The system shall render `get_goal`, `create_goal`, and `update_goal` compact as single-line goal summaries showing the objective or requested status plus current goal status, with no compact metadata rows. Expanded, it shall show objective, status, token/time accounting when present, automation/accounting-pending state when present, and the tool-result message, omitting absent/null/false fields unless false is the meaningful state, with no raw JSON. ^render-go01
- The system shall render `ralph_continue` and `ralph_finish` compact as single-line task summaries showing task id plus iteration/status facts. Expanded, it shall show task id, iteration, status, reflection flag when true, max-iteration/reflection cadence when present, and the result message, with no raw JSON. ^render-ra01
- The `notification` renderer shall not parse XML, infer success or failure from unstructured content, color by exit status, or extract output/error blocks; when a notification carries structured run status, the renderer shall reflect that status in the header dot. ^render-nt02
- The `notification` renderer shall remain registered as a custom renderer and shall render completion availability with the compact tool-like header and expandable plain-text body, flush and without a background box so its chrome matches self-shell tool rendering. ^render-nt03
- Completion-availability messages shall use `customType = "notification"`, not `customType = "taumel.notification"`. ^render-nt04
- When an exec completion notification is expanded, the system shall show the session identifier and outcome as labeled fields and shall not render model-facing consumption instructions. ^render-sj0j
- Taumel `skill` custom messages shall look exactly like Pi's standard skill invocation rendering. When Pi exposes the standard skill parser/renderer as a public package-root API, Taumel shall reuse that API rather than maintaining a visually divergent copy; if reuse becomes unavailable, Taumel's fallback renderer shall preserve visual parity with Pi's standard compact and expanded skill rendering and shall not leak the XML transport envelope. ^render-sk01

### Exa rendering

- The system shall render `web_search_exa` and `crawling_exa` compact as one-line result-count summaries. Expanded, it shall show up to the tool's reasonable display cap with `idx · title · domain`, full URL, and a dim summary/snippet/excerpt, omitting score, cost, and author, with `crawling_exa` using an `N urls` subject and a longer per-page excerpt. ^render-xa01
- The system shall render `get_code_context_exa` compact as a single-line query summary, optionally including a response-size fact when available. Expanded, it shall render the full `response` as plain human-readable output, with no raw JSON envelope. ^render-xa02
- The system shall render `exa_agent_create_run` and `exa_agent_get_run` compact as single-line run summaries showing id and status. Expanded, it shall show id, status, created/updated timestamps when present, output text when present, error when present, and other useful user-facing run metadata, with no raw JSON. `exa_agent_cancel_run` shall render compact as a single-line cancellation action. ^render-xa03
- The system shall render `exa_agent_list_runs` and `exa_agent_list_events` compact as one-line count summaries, e.g. `recent runs (N)` or `<run-id> (N events)`. Expanded, it shall list runs/events with user-facing fields such as id, status, title/type, timestamp, and summary when present, with no raw JSON and no noisy internal/cost fields unless they are the requested payload. ^render-xa04
